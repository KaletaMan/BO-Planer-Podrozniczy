```latex
\section{Eksperymenty}
\label{sec:eksperymenty}

Celem eksperymentów jest ocena jakości tras generowanych przez zaimplementowane
algorytmy oraz analiza wpływu parametrów algorytmów rojowych na jakość
rozwiązania, szybkość zbieżności oraz stabilność wyników.

W aplikacji zaimplementowano cztery metody generowania tras:
\begin{itemize}
    \item \texttt{Random Walk},
    \item \texttt{Greedy attractions},
    \item algorytm sztucznej kolonii pszczół (\texttt{Artificial Bee Colony}),
    \item algorytm inspirowany tańcem pszczół (\texttt{Bee Algorithm}).
\end{itemize}

Dodatkowo aplikacja umożliwia zapisywanie wyników do plików \texttt{JSON},
wizualizację tras oraz analizę przebiegu działania algorytmów na podstawie
historii iteracji.

\subsection{Metodologia eksperymentów}

Eksperymenty wykonywano na automatycznie generowanych instancjach problemu
planowania trasy turystycznej. Każda instancja zawiera mapę siatkową, punkt
startowy, punkt końcowy, zbiór atrakcji oraz ograniczenia czasu i budżetu.
Dla porównania algorytmów wykorzystywano tę samą instancję problemu, dzięki
czemu wyniki były porównywalne.

\paragraph{Zmienne niezależne.}
W eksperymentach można zmieniać następujące parametry:
\begin{itemize}
    \item rozmiar mapy:
    \[
        H \times W
        \in
        \{
        10 \times 10,\,
        20 \times 20,\,
        30 \times 30
        \};
    \]
    \item liczba atrakcji na mapie;
    \item rozkład wag pól: \texttt{uniform} oraz \texttt{clusters};
    \item parametry algorytmu ABC:
    \begin{itemize}
        \item rozmiar populacji $N$,
        \item liczba iteracji $I$,
        \item limit stagnacji $L$.
    \end{itemize}
    \item parametry algorytmu Bee:
    \begin{itemize}
        \item rozmiar populacji,
        \item liczba iteracji,
        \item prawdopodobieństwo rekrutacji,
        \item udział zwiadowców,
        \item próg jakości tańca,
        \item cierpliwość porzucenia rozwiązania.
    \end{itemize}
    \item ziarno generatora liczb losowych, oznaczane jako \texttt{seed}.
\end{itemize}

\paragraph{Zmienne zależne.}
Dla każdego uruchomienia mierzono:
\begin{itemize}
    \item końcową wartość rozwiązania,
    \item koszt odwiedzonych atrakcji,
    \item całkowity czas przejścia,
    \item liczbę odwiedzonych atrakcji,
    \item długość ścieżki,
    \item iterację znalezienia najlepszego rozwiązania,
    \item liczbę wywołań funkcji celu,
    \item informację, czy rozwiązanie spełnia ograniczenia czasu i budżetu.
\end{itemize}

\subsection{Reprezentacja wyników}

Każde uruchomienie algorytmu może zostać zapisane do pliku \texttt{JSON}.
Plik wynikowy zawiera mapę wejściową oraz listę uruchomień algorytmów.

Zapisywane są między innymi:
\begin{itemize}
    \item mapa wejściowa,
    \item nazwa wybranego algorytmu,
    \item końcowa ścieżka,
    \item historia najlepszych wyników w kolejnych iteracjach,
    \item iteracja znalezienia najlepszego rozwiązania,
    \item pełna ewaluacja rozwiązania.
\end{itemize}

Przykładowa struktura pojedynczego wyniku ma postać:

\begin{lstlisting}[language=json, caption={Przykładowa struktura wyniku}]
{
  "algo": "ABC",
  "path": [[0, 0], [1, 0], [2, 1]],
  "history": [
    {"iteration": 1, "best_value": 120},
    {"iteration": 2, "best_value": 140}
  ],
  "best_iteration": 17,
  "evaluation": {
    "movement_time": 31,
    "attraction_cost": 26,
    "value_collected": 180,
    "feasible": true
  }
}
\end{lstlisting}

Pole \texttt{history} jest szczególnie istotne dla algorytmów rojowych,
ponieważ pozwala analizować przebieg procesu optymalizacji, a nie tylko
końcowy wynik.

\subsection{Wyniki przykładowego eksperymentu}

W celu wstępnej oceny działania zaimplementowanych algorytmów przeprowadzono
eksperyment porównawczy na tej samej instancji problemu. Wszystkie algorytmy
korzystały z identycznej mapy oraz tych samych ograniczeń czasu i budżetu.

Parametry eksperymentu:
\begin{itemize}
    \item limit czasu: $50$,
    \item budżet: $20$,

\subsection{Analiza zbieżności}

Dla algorytmów rojowych zapisywana jest historia najlepszej wartości znalezionej
w kolejnych iteracjach. Pozwala to analizować:
\begin{itemize}
    \item szybkość poprawy rozwiązania,
    \item moment znalezienia najlepszego wyniku,
    \item stabilność działania algorytmu,
    \item wpływ parametrów na zbieżność.
\end{itemize}

Dla każdego uruchomienia algorytmów ABC oraz Bee można wygenerować wykres:

\[
    best\_value(iteration),
\]

gdzie \texttt{best\_value} oznacza najlepszą wartość rozwiązania znalezioną
do danej iteracji. Jeżeli wykres szybko się wypłaszcza, może to oznaczać,
że dalsze zwiększanie liczby iteracji nie daje istotnej poprawy. Jeżeli
natomiast poprawy pojawiają się również pod koniec działania algorytmu,
warto rozważyć zwiększenie liczby iteracji.

\subsection{Test Wilcoxona}

W celu sprawdzenia, czy różnice pomiędzy algorytmami wynikają z rzeczywistej
poprawy jakości, a nie jedynie z losowości pojedynczych uruchomień, można
zastosować test rangowanych znaków Wilcoxona.

Test ten jest odpowiedni, ponieważ algorytmy porównywane są parami na tych
samych instancjach problemu. Dla każdej mapy zapisywany jest wynik dwóch
porównywanych metod, na przykład ABC oraz Bee Algorithm:

\[
    (x_1, y_1), (x_2, y_2), \ldots, (x_n, y_n),
\]

gdzie $x_i$ oznacza wynik pierwszego algorytmu na $i$-tej mapie, a $y_i$
wynik drugiego algorytmu na tej samej mapie.

Przyjęto następujące hipotezy:
\[
    H_0:
    \text{ mediany różnic pomiędzy wynikami algorytmów są równe,}
\]
\[
    H_1:
    \text{ mediany różnic pomiędzy wynikami algorytmów różnią się.}
\]

Dla poziomu istotności:
\[
    \alpha = 0.05
\]

wynik testu interpretowany jest następująco:
\begin{itemize}
    \item jeżeli $p < 0.05$, różnica pomiędzy algorytmami jest statystycznie istotna,
    \item jeżeli $p \geq 0.05$, brak podstaw do odrzucenia hipotezy zerowej.
\end{itemize}

Przykładowe użycie testu w języku Python:

\begin{lstlisting}[language=Python, caption={Przykładowe użycie testu Wilcoxona}]
from scipy.stats import wilcoxon

abc_results = [360, 340, 380, 355, 370]
bee_results = [185, 260, 300, 250, 280]

statistic, p_value = wilcoxon(abc_results, bee_results)

print("Statystyka:", statistic)
print("p-value:", p_value)
\end{lstlisting}

Zastosowanie testu Wilcoxona pozwala sprawdzić, czy zaobserwowane różnice
pomiędzy algorytmami są powtarzalne dla wielu instancji problemu.

\subsection{Cele dalszych eksperymentów}

Przeprowadzone eksperymenty mają odpowiedzieć na następujące pytania:
\begin{enumerate}
    \item Czy algorytmy rojowe osiągają lepsze wyniki niż baseline'y?
    \item Czy Bee Algorithm osiąga lepsze wyniki niż ABC?
    \item Jak parametry algorytmów rojowych wpływają na jakość rozwiązania?
    \item Po ilu iteracjach zwykle znajdowane jest najlepsze rozwiązanie?
    \item Czy zwiększanie liczby iteracji rzeczywiście poprawia wynik?
    \item Ile wywołań funkcji celu potrzeba do znalezienia najlepszego rozwiązania?
\end{enumerate}

Dzięki zapisywaniu historii iteracji możliwe jest nie tylko porównanie
końcowych rezultatów, lecz również analiza przebiegu procesu optymalizacji.
```
